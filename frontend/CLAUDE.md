## Key files

### `lib/demoProfiles.ts`

Exports `getDemoProfiles()` (called at render time so dates stay fresh — not a static constant).

Returns 4 `DemoProfile` objects, each with:
- `payload` — a complete `ClaimCreatePayload` with hardcoded Supabase Storage URLs pointing to pre-uploaded demo PDFs
- `policyDetails` — a `PolicySummary` for the policy panel (tier, status, limits, deductible, prior claims)
- `likelihoodLevel` / `likelihoodLabel` / `likelihoodHint` — probabilistic outcome indication (not a guarantee)

Demo PDFs live under `claim-documents/demo/{scenario}/` in Supabase Storage and are generated by `backend/scripts/upload_demo_docs.py`.

### `lib/api.ts`

All backend calls. Base URL from `NEXT_PUBLIC_API_URL`.

| Function | Method | Description |
|---|---|---|
| `submitClaim(payload)` | POST `/claims` | Returns `{claim_id, status}` |
| `getClaimStatus(claimId)` | GET `/claims/{id}` | Returns `ClaimStatusResponse` |
| `validatePolicy(payload)` | POST `/claims/validate-policy` | Throws on mismatch |

`ClaimNotFoundError` is thrown by `getClaimStatus` on 404.

### `hooks/useClaimStatus.ts`

```typescript
useClaimStatus(claimId: string | null)
```

Wraps `getClaimStatus` in TanStack Query. Polls every **5 seconds** while status is `pending` or `processing`. Stops automatically on any final status. Disabled when `claimId` is null.

### `lib/storage.ts`

`uploadDocument(file, claimId, documentType)` — uploads to Supabase Storage bucket `claim-documents` at path `{claimId}/{documentType}/{uuid}.{ext}`. Returns `{file_url, file_name}`.

The storage prefix (a temp UUID) is generated on the frontend when the user reaches Step 3. Files are uploaded before `POST /claims` is called.

### `lib/documentSlots.ts`

`getSlotsForReason(reason)` — returns the list of required/optional upload slots for a given cancellation reason. Slots vary by reason (e.g., illness requires a physician statement, job loss requires a redundancy letter). The `default` slot set applies for any unmapped reason.

### `types/claim.ts`

```typescript
interface ClaimCreatePayload { /* all form fields + attachments array */ }
interface ClaimStatusResponse {
  claim_id: string
  status: 'pending' | 'processing' | 'approved' | 'rejected' | 'needs_more_info' | 'failed'
  decision_summary: string | null
  approved_amount: number | null
  created_at: string
  updated_at: string
}
```

### `lib/utils.ts`

`cn(...classes)` — clsx + tailwind-merge
`formatCurrency(amount)` — EUR format (de-DE locale)
`formatDate(dateStr)` — ISO date string → "D Month YYYY" (en-GB)

## Claim submission flow

```
Step 1: Personal details → validatePolicy() on "Next"
Step 2: Trip details + reason
Step 3: Document uploads → uploadDocument() per slot
Step 4: Review + consent → submitClaim() on submit

On success:
  → claim_id stored in sessionStorage
  → router.push("/status?id={claim_id}")
  → useClaimStatus polls every 5s until terminal status
  → ClaimResult renders the final decision
```

## Styling conventions

- **Tailwind CSS 4** — configured via `@import "tailwindcss"` in `globals.css`
- Custom utilities in `globals.css`: `.text-gradient` (indigo→violet→sky), `.bg-grid` (hero dot grid)
- Card shadow: `shadow-card` and `shadow-card-hover` (defined in globals.css)
- Rounded cards: `rounded-2xl border border-slate-200 bg-white`
- Primary buttons: `rounded-full bg-indigo-600 px-6 py-3 text-sm font-semibold text-white`
- Secondary buttons: `rounded-full border border-slate-300 bg-white px-6 py-3`
- Status colours: emerald = approved, red = rejected, orange = needs_more_info, indigo = pending/processing, amber = failed

## Adding a new page

1. Create `app/{route}/page.tsx`
2. If it needs client interactivity: add `'use client'` or extract a Client Component
3. Add a nav link to `app/layout.tsx` and `components/Footer.tsx`
4. If it polls or fetches: use `useClaimStatus` or a new TanStack Query `useQuery` inside a client component wrapped in `<Suspense>` in the server shell

## Gotchas

- **`attachments` is optional** in `ClaimCreatePayload` — can pass `[]` for demo/testing without triggering validation errors
- **Policy validation** (`validatePolicy`) checks email + date of birth against mock data — demo scenarios use the exact email/DOB from `mock_policies.json`
- **`sessionStorage`** stores the last submitted `claim_id` for the status page UX; clearing browser storage will remove this
- **Supabase Storage URLs** returned by `uploadDocument` are public — they are passed directly to the backend and Claude reads the files from these URLs natively
- **TanStack Query cache keys** for claim status use `['claim', claimId]` — each unique claim ID gets its own cache entry; no collisions between the demo page and `/status` page
